About the Pokémon Agentic RAG

1. Project Overview

This project is an advanced Retrieval-Augmented Generation (RAG) web application built to explore and query Pokemon statistics. Instead of relying on a single retrieval method, it uses an Agentic Routing Architecture to ensure high accuracy for both semantic questions and complex mathematical queries.

2. Technologies Used

  • Python & Flask - Backend architecture and web server.
  • FAISS - Vector database for fast semantic similarity search.
  • SentenceTransformers - Converting textual data into vector embeddings.
  • Pandas - Data manipulation and executing mathematical queries.
  • Google Gemini 2.5 Flash - The core LLM used for routing, and generating final human-readable answers.

3. Chunking Strategy & Embeddings

Row-Based Tabular Chunking

Unlike standard text documents that require tools like NLTK to split paragraphs, this dataset is tabular. Using standard text splitters would break the relationship between a Pokemon and its stats. Therefore, I implemented a Row-Based Chunking Strategy. Each row in the CSV was converted into a single textual string (e.g., "Name: Pikachu | Type: Electric | HP: 35...").

For the embeddings, I utilized all-MiniLM-L6-v2 from SentenceTransformers. It was chosen because it is highly efficient and performs exceptionally well on semantic search tasks for structured short texts.

4. Challenges Encountered & Solutions

The Challenge: Vector Search Limitations

A standard RAG pipeline using FAISS is excellent at finding semantic similarities. However, it struggles significantly with analytical queries like "Which Pokemon has the highest HP?" or "How many Generation 2 Pokemon exist?" because vector similarity does not understand numerical aggregation.

The Solution: Intelligent Query Routing

To solve this, I implemented an Agentic Router. When a user asks a question, an LLM evaluates the prompt first. If the question is semantic, it is routed to the FAISS pipeline. If the question requires mathematical operations, it is routed to a custom Pandas pipeline that executes dynamic code against the original dataset, guaranteeing 100% numerical accuracy.

5. Handling Edge Cases & Hallucinations

To prevent hallucinations and handle irrelevant questions (e.g., "How to bake a cake?"), the LLM is controlled by a strict system prompt. The prompt explicitly instructs the model: "If the answer cannot be found in the context... say 'I do not have enough information in the retrieved context to answer this.'" This ensures the AI acts exclusively as a Pokemon Data Expert.

6. Data Source

The dataset used in this project is based on the original Pokemon games statistics. Special thanks to the author Alberto Barradas for compiling this data.
